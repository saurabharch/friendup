Start programming

Introduction

This is a short tutorial that can get you started with developing software in FriendUP. We will go through all the main APIs by example. The rest of the knowledge you will need, is a basic understanding of HTML5 and Javascript. Information about these topics are scattered across the world wide web.

FriendUP applications are programmed both on the server side and on the client side. This tutorial will only concentrate on the client side. The reason for this is that server side programming gets complex very quickly, and the APIs that are included for the client side programming is complete enough to give you all the tools required to make an application.

FriendUP can be programmed using FriendDOS scripting, Javascript or both. In this tutorial we will only focus on Javascript - for more on FriendDOS, please see the FriendDOS documentation located in the same directory as this document.

Getting started

(Please read the documents FriendDOS and Workspace before reading this next part, as it contains information you will need.)

The first thing you should do once setting up your first FriendUP application is to organize your files. This is done with the file manager in Workspace. Double click your volume of choice, for example Home:, and make a new directory. Call it, for the sake of this document, Project1.

So for this tutorial, we will place our files in this path, with this directory structure:

Home:Project1/

  • Templates/
  • Scripts/
  • Graphics/
  • Sounds/

Once this structure is set up using the Workspace file manager, we will continue to the next step, using Artisan.

Using Artisan

Artisan is the default programmers editor in FriendUP. It supports many languages and has got features you would come to expect from a programming tool. In FriendUP, we are working with "executable Javascript", or .jsx for short. Normal Javascript files are not executable, but .jsx ones are - as they are supposed to contain a "main function".

But before we get into all that, fire up Artisan. If you want to run it from the Shell, type "launch Artisan", and hit enter. If it is the first time you are running it, it will ask you to activate it. Just accept the activation form and Artisan will launch.

We will not go too much into how Artisan works. Artisan has its own documentation, located in the System:Documentation/Applications directory. But what you want to do, is to save the empty file that opens up immediately after you have launched Artisan. Call the file Project1.jsx, and save it in the directory Home:Project1/.

Artisan can also load documents, either by using drag & drop from Workspace, or through the file dialog. You can play around a bit in Artisan to familiarize yourself. We will go on to the next step.

The run function

Application.run() is FriendUP's "main function". Unless you are familiar with C (or other programming languages of this nature), the "main function" is the function the OS, or runtime environment calls when starting a program. It is the starting point of your program.

In FriendUP, each Javascript executable is given an Application object from the system. This contains some event handling and message passing functionality, as well as some dynamic information, like applicationId and authId (which we will cover later).

Your first application should look something like this:

Application.run = function( msg )
{
    console.log( 'Hello world' );
}

At this point, it would be a good idea to get your Google Chrome debug console up. You can do this with hitting F12. (TODO: Update with using FriendDOS for log output...)

You can run the .jsx executable directly from Artisan. Click on the "play" button on the bottom of the Artisan window. It will output "Hello world" to the Google Chrome console.

When you have tested running your first .jsx program, you will have to stop the program from running. You don't want to fill up memory with pointless programs!

To check if your program is still running, you can check the Deepest Field by pulling on the Workspace screen bar on the top of the screen. Here you will see all your running tasks, and you will be able to quit the tasks by clicking on their box.

Of course, if you know that your program is running, just click the "stop" button next to the "play" button. This will stop the running program immediately.

The Application.run() function takes two arguments. The first argument is msg, which is an object that holds information relevant to the application. The second argument, iface will tell you which interface is asked for by the system. (At this time, this is not implemented). This argument is a string.

Using views

In FriendUP, desktop windows are called views. This is because a window in FriendUP can be displayed in different ways, but it always is used to view graphical user interfaces.

To open up a view in your .jsx executable, type in this code in your run function, after your console output.

var v = new View( {
    title: "My Project Number One",
    width: 640,
    height: 480
} );

v.onClose = function()
{
    Application.quit();
}

this.mainView = v;

The first part of this code opens up a new view, with the title "My Project Number One". The view will have a width of 640 pixels and a height of 480 pixels.

The second part of this code tells the Application object to quit the program if the view is closed.

The last bit of the code assigns the view object to a variable anchored to the Application object, for safe keeping (assuming you now placed this piece of code inside the Application.run() function).

Now try executing your application with the "play" button again. Then "stop" it. You have now opened and closed your first view in FriendUP.

Adding some GUI content

At this point, you would want to see something inside of your view that would resemble some kind of GUI. For this, we will introduce you to how we apply HTML5 templates as GUI in FriendUP. Additionally, we need to teach you how to use the File class, and how to use relative paths.

To start, make a new, empty file from the Artisan File menu. Save the file in your Templates/ directory with the name, "gui.html" (without the quotes of course).

Enter this piece of HTML inside the gui.html file:

<div class="Padding">
    <button onclick="Quit()" class="Button">Close</button>
</div>

<script>
    function Quit()
    {
        Application.sendMessage( { command: "quit" } );
    }
</script>

Underneath the view code you just wrote in your code file Project1.jsx, type in this piece of code:

var f = new File( "Progdir:Templates/gui.html" );
f.onLoad = function( data )
{
    Application.mainView.setContent( data );
}
f.load();

The gui.html code is the GUI for our view. Embedded in the GUI, we have a Quit() function that hooks up to the Application object, automatically created for the template scope. Quit() uses a standard Application.send method, which sends an object with the "quit" command to the Application object in Project1.jsx. This "quit" command is automatically handled by the Application object, and it immediately quits.

The GUI is fetched by opening up the gui.html file from Progdir:Templates/gui.html. Progdir: is a relative path to where the executable is located. This makes it easier to move your .jsx projects around, without having to worry about volume and directory names changing.

Try to run the program now. It will open up a new view, with a single button inside some padded margins, and will allow you to quit the program by clicking the Close button as presented. You have now written your first functional FiendUP program.

Using pulldown menus

FriendUP is menu driven and uses pulldown menus wherever possible to unclutter the GUI. The pulldown menus work on all interface modes (mobile, TV, WIMP), but are adapted in look and feel.

To add menus to your FriendUP application, add this code:

v.setMenuItems( [
    {
        name: 'File',
        items: [
            {
                name: 'Quit',
                command: 'quit'
            }
        ]
    }
] );

This code adds a File menu with one menu item, Quit. The Quit menu item sends a 'quit' command to the Application object. The 'quit' command is built-in.

When adding your own commands, you catch them in a function called Application.receiveMessage. This function takes one argument, msg, which is an object holding information about the message. The command itself is passed in msg.command, which is a string.

To show this, consider this menu, with our first custom menu item that we need to build support for ourselves:

v.setMenuItems( [
    {
        name: 'File',
        items: [
            {
                name: 'About our test',
                command: 'about'
            },
            {
                name: 'Quit',
                command: 'quit'
            }
        ]
    }
] );

We now add a new function outside the Application.run function:

Application.receiveMessage = function( msg )
{
    if( !msg.command ) return false;
    switch( msg.command )
    {
        case 'about':
            showAboutWindow();
            break;
    }
}

We have now handled our first custom menu item. 'about' gets handled by the function, showAboutWindow(), which we have to make. It should test if the about window is open, and if not, then it should open it.
